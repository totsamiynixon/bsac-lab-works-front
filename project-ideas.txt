полный workflow может быть такой
есть 5 веток: master_front, master_back, master, staging, prod
master_front содержит только файлы фронта и свой travis.yml конфиг, который ранает тесты тесты фронта
master_backend содержит только файлы бека свой travis.yml конфиг, который ранает тесты тесты бека
master содержит все файлы и свой travis.yml конфиг;
при пулл реквесте в master из master_backend или master_frontend ничего не происходит ну или можно опять запустить тесты, надо подумать
при мерже в мастер ничего не происходит, т.к. мастер нужен для того, чтобы развернуть у себя приложение целиком локально
пулл реквест master -> staging запускаются интеграционные тесты. генерируется swagger документация и запускаются интеграционные тесты фронта
мерж master -> staging деплой без тестов 
пулл реквест staging -> production возможно даже ничего - просто сигнал, что кто-то хочет задеплоить все приложение. возможно, какие-то тесты тоже для теста соединения с prod bd перед деплоем
мерж staging -> production деплой в прод
